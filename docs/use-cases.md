# Use Cases

## Сутності

- **User** — зареєстрований користувач системи
- **Admin** — користувач з роллю `admin`
- **Todo** — задача, яка належить конкретному User

---

## UC-01: Реєстрація користувача

**Актор:** Анонімний відвідувач  
**Ендпоінт:** `POST /auth/`

**Передумови:**
- Користувач не авторизований

**Основний сценарій:**
1. Користувач надсилає `username`, `email`, `password`, `first_name`, `last_name`, `phone_number`
2. Система перевіряє, що `email` є валідним (формат)
3. Система хешує пароль
4. Система зберігає користувача з роллю `user` за замовчуванням
5. Повертає `201 Created`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| `email` невалідний формат | `422 Unprocessable Entity` |
| `username` або `email` вже зайнятий | `409 Conflict` |

---

## UC-02: Вхід у систему

**Актор:** Зареєстрований User  
**Ендпоінт:** `POST /auth/token`

**Передумови:**
- Користувач зареєстрований у системі

**Основний сценарій:**
1. Користувач надсилає `username` та `password` (form data)
2. Система знаходить користувача за `username`
3. Система перевіряє пароль
4. Система генерує JWT-токен з `sub`, `id`, `role`, `exp`
5. Повертає `{ access_token, token_type: "bearer" }`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| Користувач не знайдений | `401 Unauthorized` |
| Пароль невірний | `401 Unauthorized` |

---

## UC-03: Перегляд своїх задач

**Актор:** Авторизований User  
**Ендпоінт:** `GET /todos/`

**Передумови:**
- Користувач авторизований (валідний JWT у заголовку `Authorization: Bearer <token>`)

**Основний сценарій:**
1. Система витягує `user_id` з токена
2. Повертає список усіх задач, що належать цьому користувачу
3. Статус `200 OK`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| Токен відсутній або невалідний | `401 Unauthorized` |
| Задач немає | `200 OK`, порожній список `[]` |

---

## UC-04: Перегляд окремої задачі

**Актор:** Авторизований User  
**Ендпоінт:** `GET /todos/{todo_id}`

**Передумови:**
- Користувач авторизований
- `todo_id > 0`

**Основний сценарій:**
1. Система шукає задачу за `todo_id` та `owner_id` поточного користувача
2. Повертає задачу, статус `200 OK`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| Токен невалідний | `401 Unauthorized` |
| Задача не знайдена або належить іншому користувачу | `404 Not Found` |

---

## UC-05: Створення задачі

**Актор:** Авторизований User  
**Ендпоінт:** `POST /todos/`

**Передумови:**
- Користувач авторизований

**Інваріанти:**
- `title`: 3–50 символів
- `description`: 3–200 символів
- `priority`: ціле число від 1 до 5 включно

**Основний сценарій:**
1. Користувач надсилає `title`, `description`, `priority`, `complete`
2. Система валідує поля згідно інваріантів
3. Система створює задачу з `owner_id` поточного користувача
4. Повертає `201 Created`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| Токен невалідний | `401 Unauthorized` |
| `title` коротший за 3 символи | `422 Unprocessable Entity` |
| `priority` виходить за межі 1–5 | `422 Unprocessable Entity` |

---

## UC-06: Оновлення задачі

**Актор:** Авторизований User  
**Ендпоінт:** `PUT /todos/{todo_id}`

**Передумови:**
- Користувач авторизований
- Задача існує та належить користувачу

**Інваріанти:**
- `title`: 3–50 символів
- `description`: 3–200 символів
- `priority`: ціле число від 1 до 5 включно

**Основний сценарій:**
1. Користувач надсилає оновлені поля
2. Система перевіряє, що задача належить поточному користувачу
3. Оновлює поля задачі
4. Повертає `204 No Content`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| Токен невалідний | `401 Unauthorized` |
| Задача не знайдена або чужа | `404 Not Found` |
| Невалідні поля | `422 Unprocessable Entity` |

## UC-12: Видалення користувача (адмін)

**Актор:** Admin  
**Ендпоінт:** `DELETE /admin/users/{user_id}`

**Передумови:**
- Користувач авторизований з роллю `admin`
- Користувач з `user_id` існує

**Основний сценарій:**
1. Система перевіряє роль `admin`
2. Видаляє користувача та всі його задачі (cascade)
3. Повертає `204 No Content`

**Альтернативні сценарії:**
| Ситуація | Відповідь |
|---|---|
| Токен невалідний | `401 Unauthorized` |
| Роль не `admin` | `403 Forbidden` |
| Користувач не знайдений | `404 Not Found` |
