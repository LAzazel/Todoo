# Аналіз: Лабораторна робота 5 - Модульний моноліт

## 1. Як визначив межі контекстів

Відправна точка: питання «що знає про що і чому». У попередніх лабораторних весь код жив в одному просторі: `create_todo.py` знав про `IAuditService`, аудит знав про `TodoEvents`, роутери знали про все підряд. Це не проблема для маленького проекту, але ознака відсутності меж.

Виділив два контексти: Core і Analytics.

**Core** - це все що було побудовано з лаби 2: todos, users, auth, admin. Єдиний критерій потрапляння сюди - чи є це частиною основної бізнес-операції. Якщо без цього система не може виконати свою головну функцію (створити задачу, змінити статус, зареєструвати користувача) - це Core. Аудит залишився тут же, бо він записує факти безпосередньо в процесі операції і не є окремим споживачем.

**Analytics** - окремий споживач, який будує власні проєкції поверх подій Core. Він не бере участі в жодній бізнес-операції. Якщо Analytics впаде - `POST /todos` продовжить працювати. Якщо Analytics відстане на кілька секунд - нічого страшного: статистика eventual, і це прийнятно бо ніхто не приймає бізнес-рішення в реальному часі на основі лічильника todos.

Альтернативи які розглядав: виділити Users в окремий контекст або Auth як Identity context. Відмовився бо Users і Todos сильно пов'язані через `owner_id` - розрив цього зв'язку дав би штучний поділ заради поділу. Auth ще менше кандидат: це просто механізм доступу, а не окрема бізнес-область.

## 2. Модульна структура

Кожен модуль отримав повну внутрішню архітектуру: `domain/`, `application/`, `infrastructure/`, `presentation/`. Це дзеркалить шарову архітектуру з лаби 2, але тепер кожен модуль є незалежним вертикальним зрізом.

Публічний контракт Core - `public_contract.py` - експортує тільки integration events. Інші модулі не мають права імпортувати нічого з `core/domain/`, `core/application/` чи `core/infrastructure/` напряму. Це єдина точка входу.

Analytics теж отримав `public_contract.py` з `get_stats_store()`, щоб `main.py` не лізло у внутрішні файли модуля.

## 3. ACL і трансляція моделей

`AnalyticsACL` отримує `TodoCreatedIntegrationEvent` з публічного контракту Core і перекладає його у `TodoActivityRecorded` - власну внутрішню модель Analytics. Ключовий момент: `TodoCreatedIntegrationEvent` містить `priority: int`, але Analytics не знає що таке `Priority` як value object - він знає тільки `is_high_priority: bool`. Це і є суть ACL: зовнішня мова не проникає всередину.

Якщо Core завтра перейменує `priority` у `urgency_level` або змінить шкалу з 1–5 на 1–10 - зміниться тільки `acl.py` в Analytics. Жоден з handlers, domain models чи projections не зачепиться.

Те ж саме для `UserRegisteredIntegrationEvent`: Analytics отримує факт реєстрації і просто інкрементує лічильник, не зберігаючи `user_id`, `username` чи `email`.

## 4. Міжмодульна комунікація через події

Integration events публікуються з command handlers в Core одразу після збереження основної сутності. Analytics підписується на ці events через `InMemoryEventBus` і обробляє їх синхронно в тому ж процесі - але семантично це eventual consistency: основна операція вже завершена і повернула результат до того як підписник відпрацює.

Де strong consistency: все всередині одного модуля Core. Транзакція в SQLite, аудит запис, доменні інваріанти - все це відбувається як одна неподільна одиниця з точки зору клієнта.

Де eventual consistency: між Core і Analytics. `SystemStats` оновлюється асинхронно після того як HTTP-відповідь вже пішла. Якщо підписник кине exception - `InMemoryEventBus` його проковтне і статистика не оновиться. Для лічильника todos це прийнятно: втрата одного increment не є бізнес-проблемою. Для фінансових транзакцій - ні.

Такий розподіл прийнятний бо Analytics - read-only observability-модуль. Він нікого не блокує і не є частиною критичного шляху.

## 5. Що зміниться якщо виділити Analytics в окремий сервіс

Поточний `InMemoryEventBus` замінюється на брокер повідомлень - RabbitMQ або Kafka. Core публікує integration events у топік, Analytics підписується з окремого процесу. `SystemStats` переїжджає з пам'яті в PostgreSQL. `AnalyticsACL` залишається без змін - він вже ізольований. `public_contract.py` Core стає схемою повідомлень. Публічний контракт Analytics стає REST.

Ключове: жодна зміна не торкнеться Core. Це і є головна цінність модульного моноліту - межі вже проведені, і вони готові до фізичного розриву.

## Ретроспектива курсу: від лаби 1 до лаби 5

**Лаба 1**. Плаский FastAPI-додаток. Роутери напряму звертались до SQLAlchemy, бізнес-логіка жила в Pydantic-схемах і функціях роутерів. Просто і швидко, але будь-яка зміна вимагала йти в декілька місць одночасно.

**Лаба 2**. Шарова архітектура. Поява Domain, Application, Infrastructure, Presentation. Найбільший стрибок по складності і найбільший виграш по тестованості. З цього моменту домен можна тестувати без БД і FastAPI. Головне рішення яке все визначило далі - Rich Domain Model з value objects і фабриками.

**Лаба 3**. CQS. Розбиття на `commands/` і `queries/` зробило наміри явними. Додати нову операцію - новий файл, нічого існуючого не чіпається. Read models відокремились від доменних моделей.

**Лаба 4**. Асинхронна комунікація через events і event bus. Поява аудиту як побічного ефекту. Хендлери перестали знати про конкретних споживачів, почали публікувати факти.

**Лаба 5**. Модульний моноліт. Events з лаби 4 стали integration events між модулями. Уся система залишилась одним артефактом, але всередині з'явились чіткі межі і публічні контракти.

**Найціннішим рішенням** виявилась шарова архітектура з лаби 2 - вона зробила всі наступні трансформації безболісними. Rich Domain Model з value objects і абстрактними репозиторіями дала стабільне ядро яке не змінювалось від лаби до лаби. Якби в лабі 1 залишились `db.query()` прямо в роутерах - кожна наступна лаба була б рефакторингом з нуля.

**Що зробив би інакше**: в лабі 4 одразу б розділив domain events і integration events - зараз є і `TodoCreated` (domain event) і `TodoCreatedIntegrationEvent` (integration event) які існують паралельно без чіткого зв'язку між собою. Краще було б зробити integration events похідними від domain events через mapper, а не окремими класами з іншим набором полів.

**Головний trade-off курсу**: кожна лаба додавала складність заради гнучкості. Лаба 1 - 5 файлів, лаба 5 - 60+. Для todo-додатка це явний overkill. Але саме на такому простому домені легко побачити архітектурні патерни без когнітивного навантаження від складної предметної області. Реальна цінність не в самих патернах, а в розумінні коли їх застосовувати, а коли - ні.
