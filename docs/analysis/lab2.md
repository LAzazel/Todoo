# Аналіз: Лабораторна робота 2 - Шарова архітектура

## 1. Що змінилося порівняно з лабораторною 1?

У лабораторній 1 весь код жив пласко в одній папці: роутери напряму звертались до `db.query(...)`, валідація була в Pydantic-схемах, JWT-логіка - в окремому файлі без жодної абстракції. Бізнес-правила були розсіяні по роутерах і моделях без чіткої межі.

У лабораторній 2 код розбито на 4 шари: **Domain** (моделі, value objects, інтерфейси, фабрики), **Application** (use cases, DTO), **Infrastructure** (SQLAlchemy, маппери, JWT), **Presentation** (FastAPI роутери). Кількість файлів зросла, але кожен має одну відповідальність.

## 2. Які переваги дало розділення?

Головне - тестованість. Domain і Application тестуються без БД і HTTP-сервера за мілісекунди. Зміна формату токена торкається тільки `jwt_service.py`, зміна бізнес-правила - тільки відповідного value object чи фабрики. Доменні помилки не знають про HTTP - це знання сконцентровано в одному місці (`error_handler.py`).

## 3. Які недоліки з'явились?

Для CRUD-операції тепер потрібно пройти 5 файлів замість одного. Маппери (`to_domain`, `to_orm`) - додатковий код який треба тримати синхронізованим. Новому розробнику потрібно більше часу щоб розібратися в структурі. Для невеликого проекту це відчутний overhead.

## 4. Наскільки простіше змінити БД або фреймворк?

Заміна SQLite на PostgreSQL - один рядок у `.env`. Заміна SQLAlchemy на інший ORM - тільки файли в `infrastructure/`, домен і application не торкаються. Заміна FastAPI - тільки `presentation/` і `main.py`. У лабораторній 1 заміна ORM зачепила б кожен роутер.

## 5. Чому обрано Rich Domain Model?

Обрано свідомо, бо хотілось спробувати більш професійний і масштабний підхід, а не обмежуватись простими dataclass без логіки. Rich Domain Model читається природно, інваріанти захищені самим об'єктом, а не зовнішнім кодом. Для реального продакшн-проекту з розростаючимся доменом цей підхід виграє - логіка залишається при об'єкті, а не розповзається по use cases.
