# Аналіз: Лабораторна робота 3 - CQS

## 1. Що змінилося порівняно з лабораторною 2?

У лабораторній 2 всі сценарії жили в `use_cases/` і кожен клас міг і читати і писати. Різниця між `CreateTodoUseCase` і `GetTodoUseCase` була лише в назві.

У цій лабораторній `use_cases/` повністю замінив на дві папки: `commands/` для всього що змінює стан, і `queries/` для всього що читає. Також з'явились `read_models/` - прості frozen dataclass-и які повертаються клієнту замість доменних моделей.

## 2. Переваги CQS

Найбільше сподобалось те, що тепер одразу видно що робить кожен файл. `CreateTodoCommand` - очевидно пише, `GetAllTodosQuery` - очевидно читає. Раніше треба було заходити всередину щоб зрозуміти.

Ще один плюс - простіше додавати нову функціональність. Нова команда це просто новий файл, який ніяк не торкається існуючих. У лабі 2 не завжди було зрозуміло куди додавати новий метод.

## 3. Недоліки CQS

Файлів стало помітно більше. 8 команд плюс 5 запитів замість ~8 use cases - для такого маленького проекту це виглядає як багато коду заради коду.

Є також неоднозначний момент з логіном. `LoginQuery` повертає токен - формально це читання, але з side effect у вигляді генерації токена. Чисто по CQS це спірно, але альтернатив без ускладнення коду небагато.

## 4. Command/Query Handler vs Service

У лабі 2 один UseCase міг мати кілька методів або відповідати за кілька речей одночасно. Handler по CQS - це завжди один клас, один метод `execute()`, одна відповідальність. Це робить кожен клас маленьким і легким для розуміння. Змінюєш один Handler - точно знаєш що більше нічого не зламаєш.

## 5. Вплив на розширюваність

Додати нову команду зараз це буквально: створити файл, написати Command і Handler, зареєструвати в `dependencies.py`. Жоден існуючий файл не чіпається. У лабі 2 треба було лізти в існуючий use case або придумувати куди додати новий клас.

## 6. Read Model vs доменна модель

Доменна модель `Todo` містить `Priority` як value object і методи типу `mark_as_completed()`. Це об'єкт з поведінкою і захистом інваріантів. `TodoReadModel` - просто `priority: int` і ніяких методів.

Це важливо тому що клієнту не потрібна бізнес-логіка, йому потрібні дані. Якщо завтра треба додати нове поле у відповідь або прибрати зайве - Read Model міняється незалежно від доменної моделі. Жодних ризиків зламати бізнес-правила.
